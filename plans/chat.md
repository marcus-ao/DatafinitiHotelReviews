## 1. 课题一句话定义

本课题拟构建一个**面向酒店推荐场景的对话式推荐系统原型**：以酒店评论为数据基础，先抽取“方面—情感—证据”形成细粒度知识库，再通过结构化筛选与 RAG 检索提供可验证证据，最后由经过 PEFT/QLoRA 轻量微调的小语言模型完成偏好理解、澄清交互和推荐解释生成。

由于该系统的流程基本固定、状态需要持续维护，因此在工程实现上更适合被定义为**有状态推荐工作流**而不是开放式自主 agent；LangGraph 官方也明确将其定位为适合构建长生命周期、可持久化、可调试的有状态 workflow/agent 基础设施，而 LangGraph 文档同时区分了“workflow 的预定义流程”与“agent 的动态决策流程”。

## 2. 课题核心目标

### 2.1 总体目标

在有限算力条件下，设计并实现一个能够完成以下任务的酒店推荐系统原型：

1. 理解用户自然语言需求；
2. 在酒店评论中检索与需求相关的证据；
3. 依据证据生成可解释推荐；
4. 在多轮对话中通过澄清与反馈逐步修正推荐结果；
5. 通过对比实验验证两条研究问题。

### 2.2 两个核心研究问题

#### RQ1：细粒度方面知识库是否比原始评论直接 RAG 更能提升推荐解释的可信度与可验证性？

这个问题关注的重点不是“推荐文案是否更流畅”，而是**知识组织方式**是否会影响证据检索质量、解释 groundedness 和用户对推荐可信度的感知。RAGAs 明确指出，RAG 系统评估应同时考虑检索上下文质量、回答忠实度以及最终输出质量；LangSmith 的 RAG 评测文档也将 retrieval quality、answer relevance、answer accuracy 等维度分开看待。

#### RQ2：PEFT 微调对会话式推荐的主要收益是否体现在偏好解析和澄清交互，而非事实性知识？

这个问题要验证的是：当酒店事实主要由知识库和检索模块提供时，PEFT 的主要价值是否体现在**用户需求结构化、澄清提问、约束遵守、证据式表达**等行为能力，而不是“把酒店知识记进模型参数”。

## 3. 课题总体思路：把系统“拆开做”

你现在最需要建立的不是“多堆几个热门名词”，而是一个非常清晰的系统分层观：

### 3.1 三层系统结构

#### 第一层：事实层（知识库）

负责从评论中沉淀出**可检索、可统计、可验证**的事实依据，包括：

* 酒店基础信息；
* 评论原文与句子切分结果；
* 方面类别；
* 方面情感极性；
* 代表性证据句；
* 向量索引。

#### 第二层：决策层（筛选与检索）

负责把用户自然语言需求转换成可执行约束，并进行：

* 结构化过滤；
* 候选酒店排序；
* 证据句召回与重排。

#### 第三层：交互层（LLM）

负责：

* 偏好解析；
* 澄清交互；
* 证据约束下的推荐生成；
* 多轮反馈后的约束更新。

## 4. 本课题的最终系统定位

### 4.1 系统不是“纯聊天机器人”

它的本质是一个**任务型、推荐型、证据驱动的对话系统**。

### 4.2 系统也不是“纯推荐模型”

它不是只给一个酒店列表，而是要做到：

* 能理解用户约束；
* 能说明推荐理由；
* 能解释证据来源；
* 能根据后续反馈继续调整推荐。

### 4.3 最准确的工程定位

建议你在论文里将其表述为：

> **基于细粒度评论知识库、PEFT 微调语言模型与 RAG 机制的有状态会话式酒店推荐工作流**

这样既保留“推荐智能体”的外部说法，又避免“agent”定义过宽带来的实现压力。

## 6. 核心设计原则

### 6.1 知识外置，能力内化

这是整个课题最关键的设计原则。

* **知识库 / RAG 负责事实**
* **微调模型负责行为能力**

也就是说：

* 酒店是否安静、卫生如何、地理位置怎么样，这些信息尽量放在外部知识库里；
* 模型负责“怎么理解用户需求、怎么问问题、怎么组织证据、怎么生成解释”。

这样做的原因是：评论数据动态、噪声大、更新快，外置知识更便于更新与验证；而 PEFT 天然更适合做行为适配。

### 6.2 先做“可控系统”，再做“更智能系统”

毕业设计最怕一开始追求“全自动、全开放、全智能”，结果系统不可控。
因此建议你优先做：

* 封闭方面体系；
* 明确约束解析；
* 有控制的证据生成；
* 固定流程的多轮交互。

### 6.3 实验优先于炫技

本课题最终能否成立，不在于你用了多少框架，而在于是否能通过实验回答那两个研究问题。

## 7. 各模块完整设计

---

## 7.1 模块一：细粒度方面知识库构建

### 7.1.1 模块目标

从原始评论中构建一个兼具**结构化统计能力**与**语义检索能力**的酒店评论知识底座。

### 7.1.2 数据输入

建议数据来源分两类：

#### A. 主数据源

优先使用公开酒店评论数据集，保证合法合规、结构稳定、便于复现。

#### B. 补充数据源

若条件允许，在合规前提下补充中文 OTA 场景数据，但不建议把爬虫部分作为论文重点。

### 7.1.3 预处理流程

1. 去 HTML、表情、广告噪声；
2. 删除过短或信息量极低的评论；
3. 按句号、分号、感叹号等进行中文分句；
4. 统一酒店主键；
5. 规范价格、区域、星级等字段。

### 7.1.4 方面抽取策略

PyABSA 是一个面向 ABSA 任务的开源框架，支持 APC、ATEPC、ASTE 等任务，并提供训练、推理、增强和可视化能力。对本课题而言，它适合承担方面级情感分析的实验入口。

但从毕业设计风险控制角度，**不建议一开始就追求完全开放式三元组抽取**。
更推荐的做法是：

### 7.1.5 建议采用“封闭方面体系”

先定义 6–8 个固定方面：

* 位置交通
* 卫生环境
* 服务质量
* 设施舒适度
* 性价比
* 安静程度
* 亲子友好
* 商务适配

然后执行两步：

1. 评论句映射到一个或多个方面；
2. 对每个方面判定极性（正向 / 负向 / 中性）。

这样比直接做开放式 ASTE 更稳，更容易人工校验，也更适合后续推荐排序与统计分析。这个思路和 PyABSA 所支持的任务路径并不冲突，只是你在研究设计上选择更可控的落地方式。

### 7.1.6 知识库存储结构

建议至少建立以下 4 张核心表：

#### hotel 表

* hotel_id
* 酒店名称
* 城市
* 区域
* 星级
* 价格区间
* 品牌
* 酒店简介

#### review 表

* review_id
* hotel_id
* 评论时间
* 评分
* 原始评论文本
* 清洗后文本

#### aspect_sentiment 表

* hotel_id
* review_id
* sentence_id
* aspect_category
* sentiment_label
* opinion_span
* confidence

#### evidence_index / vector_store

* text_id
* hotel_id
* sentence_text
* aspect_category
* embedding
* metadata

### 7.1.7 向量检索方案

BGE v1.5 系列模型提供中文 embedding 能力，而其官方模型卡同时明确建议：embedding 模型先召回后，再使用 `bge-reranker-base` 或 `bge-reranker-large` 对 top-k 结果进行重排，因为 reranker 通常比纯 embedding 更强。

因此推荐配置为：

* 召回模型：`BAAI/bge-small-zh-v1.5` 或 `bge-base-zh-v1.5`
* 重排模型：`bge-reranker-base`

### 7.1.8 向量库选型

Faiss 官方文档将其定义为用于高效相似度搜索与聚类的 dense vector 检索库，支持大规模向量搜索与参数调优。对毕业设计这类中等规模实验而言，Faiss 很适合作为实验阶段的向量检索基础设施。([Faiss][6])

### 7.1.9 该模块的最终输出

这一模块最终应输出：

1. 酒店结构化信息表；
2. 评论与分句表；
3. 方面情感表；
4. 方面聚合统计表；
5. 向量索引；
6. 可用于 RAG 的证据片段集合。

## 7.2 模块二：用户偏好解析与澄清交互

### 7.2.1 模块目标

把自然语言需求转成可执行约束，并判断当前信息是否足够支持推荐。

### 7.2.2 推荐抽取的槽位

建议固定以下槽位：

* 城市
* 区域 / 商圈
* 预算上限 / 区间
* 入住时间
* 出行目的（出差 / 旅游 / 亲子 / 情侣等）
* 关注方面（安静、卫生、地铁近等）
* 硬约束
* 软偏好
* 不接受项

### 7.2.3 澄清交互的设计原则

当以下信息不足时，系统进入澄清模式：

* 城市缺失；
* 预算缺失；
* 区域与价格冲突明显；
* 用户表达过于模糊；
* 条件无法形成可执行候选集。

### 7.2.4 为什么这一块要微调

这部分最适合用 PEFT 做任务适配，因为它更像“行为模式学习”而不是“知识记忆”。PEFT 文档说明 LoRA 是通过在注意力等层中引入低秩适配矩阵来做参数高效微调；PEFT 的量化文档则说明结合 4-bit 量化与 LoRA，可以在更低资源条件下训练较大模型。

### 7.2.5 输出格式

建议这一步先输出 JSON，而不是直接输出自然语言，例如：

```json
{
  "city": "上海",
  "budget_max": 400,
  "trip_type": "出差",
  "focus_aspects": ["安静程度", "位置交通"],
  "hard_constraints": ["近地铁"],
  "clarification_needed": false
}
```

这样便于调试，也便于后续做偏好解析实验。

## 7.3 模块三：候选酒店筛选与排序

### 7.3.1 模块目标

不要让 LLM 直接从全量评论中“自由发挥”，而是先通过规则和统计方法缩小候选空间。

### 7.3.2 三级筛选流程

#### 第一步：结构化过滤

根据以下硬约束过滤：

* 城市
* 区域
* 预算
* 酒店类型

#### 第二步：方面加权排序

根据用户关注方面计算综合分，例如：

* 安静程度：0.35
* 卫生环境：0.25
* 服务质量：0.20
* 位置交通：0.20

#### 第三步：证据召回

对排序靠前的 Top-N 酒店，再从评论句向量库中召回最相关证据。

### 7.3.3 建议加入两个“很加分”的统计特征

#### 时间衰减

新评论权重更高，体现近期体验。

#### 争议度

若同一方面正负评论分歧很大，则视为“争议高”。

这会让系统更像一个真实推荐系统，而不是简单情感统计器。

---

## 7.4 模块四：RAG 证据检索与上下文组装

### 7.4.1 模块目标

从候选酒店中找出最能支持推荐理由的证据，而不是简单堆评论。

### 7.4.2 推荐的检索流程

1. 结构过滤出候选酒店；
2. 对候选酒店的评论句做 dense retrieval；
3. 用 BGE reranker 对 Top-K 证据重排；
4. 选取 3–8 条高质量证据进入上下文。

BGE 官方模型卡明确建议 embedding + reranker 的组合。

### 7.4.3 上下文组织方式

建议分为三部分：

#### A. 用户需求摘要

结构化约束转写成简洁说明。

#### B. 候选酒店摘要

每家酒店给出：

* 价格
* 区域
* 主要优点
* 主要风险

#### C. 证据片段

每条证据附带：

* 酒店名
* 方面类别
* 评论时间
* 评论句内容

## 7.5 模块五：LLM 推荐生成与多轮更新

### 7.5.1 模块目标

在给定候选酒店与证据的情况下，输出：

* Top-N 推荐；
* 对每家的推荐理由；
* 风险提示；
* 当证据不足时的谨慎说明。

### 7.5.2 模型选型建议

Qwen2.5 有 3B、7B 等多个指令模型版本；Qwen3-4B-Instruct-2507 官方模型卡显示其参数量为 4.0B，原生支持 262,144 上下文长度，并强调 instruction following、general capabilities 等能力。对你的任务而言，3B/4B 级模型更适合作为主实验模型，7B 可作为增强版对照。

### 7.5.3 为什么优先选择小模型

原因不是“小模型一定最好”，而是：

* 你的任务是受约束推荐，而不是开放世界复杂推理；
* 检索与候选排序对最终效果影响很大；
* 小模型更适合毕业设计快速迭代与多轮实验。

### 7.5.4 微调方式建议

采用 QLoRA：

* 4-bit 量化加载模型；
* 训练 LoRA 适配器；
* 优先做行为微调而非知识灌输。

Transformers 与 PEFT 文档都表明 4-bit/bitsandbytes 是当前较成熟的量化路径，而 PEFT 文档也指出量化 + PEFT 是单 GPU 训练大模型的典型策略。

### 7.5.5 建议微调的任务类型

你的 SFT/PEFT 数据不要只做“用户需求 -> 最终推荐文案”，而应该覆盖：

1. 偏好解析样本
2. 澄清提问样本
3. 证据式推荐样本
4. 约束遵守样本
5. 证据不足时的谨慎回复样本

### 7.5.6 输出要求

最终输出建议始终遵守：

* 不虚构酒店；
* 不虚构设施与地理细节；
* 推荐理由必须可被给定证据支持；
* 证据不足时显式说明不确定。

## 8. 整体编排框架建议

### 8.1 为什么不建议直接用“开放式 Agent”

因为你的系统流程高度可定义：

* 输入需求；
* 判断是否需要澄清；
* 解析槽位；
* 候选过滤；
* 证据检索；
* 推荐生成；
* 接收反馈；
* 重新执行。

这更接近一个**固定状态机**，而不是任意工具调用的 agent。

### 8.2 推荐编排方案

* **LangGraph**：主流程编排与状态管理
* **LangChain 组件**：Prompt、retriever、LLM 调用、memory 工具化

LangGraph 官方说明其适合构建有状态、可持久化、可调试的长生命周期工作流；这和你的多轮推荐任务是契合的。

## 9. 两个创新点如何落到实验上

---

## 9.1 创新点一：细粒度方面知识库 vs 原始评论直接 RAG

### 研究问题

细粒度方面知识库是否比原始评论直接 RAG 更能提升推荐解释的可信度与可验证性？

### 对应实验比较

* **Raw Review RAG**：直接从原始评论句中召回
* **Aspect-KB RAG**：先做方面归一化与方面情感组织，再做召回与证据组合

### 主要观察指标

* Recall@K
* nDCG@K
* Evidence Relevance
* Citation Precision
* Citation Coverage
* Groundedness / Faithfulness
* Hallucination Rate
* 用户主观可信度

RAGAs 与 LangSmith 的评估框架都支持你把该问题拆成“检索质量 + groundedness + 最终输出质量”三个层面。([ACL Anthology][2])

### 预期结论方向

若 Aspect-KB RAG 在检索相关性、证据覆盖率和 groundedness 上明显更优，就可以支撑你的第一条创新点。

---

## 9.2 创新点二：PEFT 的收益主要体现在会话行为

### 研究问题

PEFT 微调对会话式推荐的主要收益，是否体现在偏好解析和澄清交互，而非事实性知识？

### 对应实验比较

* **Base-LLM**
* **PEFT-LLM**

### 建议拆成三个子任务

#### A. 偏好解析

输入用户话语，输出结构化槽位。

#### B. 澄清提问

判断该不该问、问什么。

#### C. 固定证据条件下的推荐生成

给 Base 和 PEFT 模型相同的候选酒店和相同证据，比较它们：

* 约束遵守能力；
* 表达组织能力；
* groundedness；
* 自然度。

### 预期结论方向

若 PEFT 在 Slot F1、澄清质量、约束遵守、自然度等方面显著提升，但在固定证据条件下对“事实正确率”的提升不明显，那么第二条创新点就能成立。

CRS 研究综述和关于 LLM 时代 CRS 评估的论文都指出，单纯比较“是否命中标准推荐项”是不够的，会低估系统在交互能力和解释性上的价值。

## 一、对你原方案的关键修改

### 1）把“手动爬网页”改成“官方 API 优先，网页采集兜底”

这是你整个项目里最值钱的一次优化。
建议顺序是：

**Plan A：飞猪开放平台 API。**
先用城市接口拿城市码，再用酒店详情接口拿酒店基础信息，再用评论接口拿评论。酒店详情接口返回酒店名、地址、城市、星级、评分、评论数、描述、设施、品牌等；评论接口返回评论内容、时间、总分、标题、回复、图片等。([Fliggy][1])

**Plan B：小规模网页补采。**
如果 48 小时内 AppKey/权限流程走不通，就不要继续卡住项目，把方案切成“**公开数据/已有样本作主集 + 飞猪少量案例集作补充验证**”。你的论文重点应放在**知识组织、RAG、PEFT 和评测**，而不是放在“我怎么把网页抠下来”。

### 2）把“多城市/大而全”改成“单城市、中等规模、可控变量”

一个月版本里，**只做 1 个城市**最合理。
我建议目标规模是：

* 80–120 家酒店
* 每家抓 30–50 条评论
* 总评论量 2500–5000 条
* 分句后大约 1–2 万个句子单元

这个量已经足够支撑知识库、检索、排序和 2×2 实验，而且不会把你拖死在清洗和标注上。

### 3）把 8 个方面缩成 6 个“原子方面”

你原来把“亲子友好”“商务适配”也放进方面体系，我建议改掉。
原因很简单：这两个更像**场景标签**，不是稳定的、句子级的原子方面，标注时会很飘。

更稳的 6 类核心方面是：

* 位置交通
* 卫生清洁
* 服务体验
* 设施房间
* 安静睡眠
* 性价比

而“亲子/商务/情侣/出差”改成**场景标签**，由关键词、规则或单独的小分类器去补，不要和 ABSA 主任务绑死。

### 4）把“开放式 ASTE”改成“封闭标签 + 弱监督 + 少量人工金标”

PyABSA 确实是成熟的 ABSA 框架，支持训练、推理、增强和可视化；但在你这个周期里，**不建议把“自己训一个完整开放式 ASTE 模型”设成主线**。更稳的做法是：

* 先做句子切分；
* 用规则/关键词做第一轮方面候选；
* 用 LLM 或现成 ABSA 工具做弱标签；
* 人工校验 400–600 句，形成金标 dev/test；
* 最终只保留“句子—方面—极性—置信度—证据”这条主链。([PyABSA Documentation][3])

### 5）把 LangGraph 从“必做”改成“可选加分项”

LangGraph 很适合这类状态型流程，但**对你来说它不是研究核心**。你完全可以先用：

`Python 函数 + 状态字典 + SQLite / CSV / Faiss`

把流程跑通。
如果你对 LangGraph 已经熟，再把它接进来做演示和状态编排；如果不熟，就不要让框架学习成本吞掉你 1/4 的时间。LangGraph 文档本身也在强调 workflow 与 agent 的边界，而不是逼你一定上复杂 agent。([LangChain Docs][2])

### 6）把微调目标收窄到“行为能力”，不要试图把酒店知识塞进参数

这一点你的原框架方向是对的，我建议进一步收紧：
PEFT 只负责三件事：

* 偏好解析
* 澄清提问
* 证据约束下的推荐表达

LoRA/PEFT 文档说明，LoRA 是通过低秩适配矩阵做参数高效微调；bitsandbytes 文档说明 QLoRA/4-bit 会把底模量化到 4-bit，再插入小规模可训练 LoRA 权重；PEFT 还专门给出 QLoRA 风格的 `target_modules="all-linear"` 用法。对你这种资源受限课题，这正是最合适的路径。([Hugging Face][4])

## 三、我建议你直接照着做的完整实施方案

### 1. 数据层：先把“证据底座”做扎实

数据表我建议从 4 张扩成 6 张，原因是后续评测和可追溯性会舒服很多：

**hotel**
`hotel_id, source_shid, source_item_id, city, district, star, rate_score, rate_number, brand, address, description, facilities, collect_time`

**review**
`review_id, hotel_id, gmt_create, total_score, title, raw_content, clean_content, picture_count, reply_count`

**sentence**
`sentence_id, review_id, hotel_id, sentence_text, sent_order`

**aspect_label**
`sentence_id, aspect, sentiment, confidence, label_source, evidence_span`

**hotel_aspect_profile**
`hotel_id, aspect, pos_cnt, neg_cnt, neu_cnt, recency_score, controversy_score, final_aspect_score`

**evidence_index**
`text_id, hotel_id, sentence_id, aspect, embedding, metadata`

这里有两个很关键的细节。
第一，飞猪评论接口文档里用的是 `item_id`，而酒店详情列表接口文档里明显返回的是 `shid`，所以你必须把 **shid ↔ item_id 映射** 单独列成采集阶段的风险项和任务项；在我检索到的页面里，这一步不是天然打通的。第二，评论接口返回了 `user_id`、`user_nick` 等字段，但你论文数据集里**不要保留这些用户身份字段**，直接丢弃或脱敏，只保留评论文本和时间等必要字段。([Fliggy][6])

### 2. 方面知识库：做“可控标签”，不做“炫技抽取”

你的核心不是证明自己能做最难的 ABSA，而是证明**“方面组织过的知识”优于“原始评论直接检索”**。
所以建议流程定成：

* 分句
* 六类方面映射
* 三分类情感（正/负/中）
* 人工校验一小部分
* 聚合成酒店方面画像

酒店方面得分可以简单写成：

`AspectScore = recent_weighted_positive - recent_weighted_negative - controversy_penalty`

其中：

* 近 90 天：1.2
* 91–365 天：1.0
* 超过 365 天：0.8

这类离散时间桶已经够用了，没必要做连续指数衰减。

### 3. 检索与排序：固定三级流程

这里不要让 LLM 直接“从全库里想”。固定成：

**第一步：结构化过滤**
城市、预算、区域、酒店类型、明显禁忌项。

**第二步：方面加权排序**
例如用户说“预算 400 以内、安静、卫生、出差、近地铁”，就把“安静”“卫生”“交通”权重拉高。

**第三步：证据召回与重排**
BGE 官方模型卡明确建议用 embedding 先召回，再用 `bge-reranker-base` / `large` 做重排；Faiss 文档则说明它就是做 dense vector similarity search 和参数调优的高效库。你的毕业设计就用：

* `BAAI/bge-small-zh-v1.5`
* `BAAI/bge-reranker-base`
* `Faiss`

这一套最稳。([Hugging Face][7])

我建议每家候选酒店最后只保留 **3–5 条证据句**，并加一个约束：
**每个方面最多 2 条证据，避免全是同一种夸奖。**

### 4. 交互层：把 LLM 输出先变 JSON，再变自然语言

这是工程上最省命的一步。
推荐你把偏好解析输出定成下面这种：

```json
{
  "city": "上海",
  "district": ["陆家嘴"],
  "budget": {"max": 400, "unit": "CNY/night"},
  "trip_type": "business",
  "focus_aspects": [
    {"aspect": "安静睡眠", "weight": 0.4},
    {"aspect": "卫生清洁", "weight": 0.3},
    {"aspect": "位置交通", "weight": 0.3}
  ],
  "hard_constraints": ["近地铁"],
  "soft_preferences": ["前台服务好"],
  "avoid": ["隔音差", "装修老旧"],
  "clarification_needed": false,
  "clarification_slots": []
}
```

澄清也不要开放问，固定成模板化问法。
例如只允许补问这 5 类槽位：

* 城市
* 预算
* 区域
* 出行目的
* 最关注方面

这样你会非常好评测。

### 5. PEFT 微调：只训 3 类样本

训练集不要铺太大。一个月里最合理的是 **800–1200 条 SFT 样本**，其中：

* 400 条偏好解析
* 200 条澄清判断/澄清生成
* 250 条证据约束推荐
* 150 条反馈更新

训练方式就用：

* 4-bit quantization
* LoRA / QLoRA
* 2–3 epoch
* 小 batch + gradient accumulation

PEFT/LoRA 与 bitsandbytes 文档都支持这条路线；Qwen 小模型尺寸也足够你完成这个任务。([Hugging Face][8])

最关键的一条是：
**推荐生成训练样本必须带证据。**
不要训“用户需求 → 空口推荐文案”，而要训：

`用户需求 + 候选摘要 + 证据句 → 推荐解释`

这样 RQ2 才能被干净地回答。

### 6. 多轮范围：只做“两次更新”

不要做无限对话。
你的多轮流程最多做到：

* Turn 1：用户初始需求
* Turn 2：系统澄清
* Turn 3：用户补充后推荐
* Turn 4：用户反馈“太贵/我更想住地铁旁/不要老酒店”
* Turn 5：系统更新约束后再次推荐

这已经足够了。答辩时你完全可以说系统支持多轮状态更新，但当前论文聚焦于**单次澄清 + 单次反馈修正**。

## 四、实验怎么改，才能既完整又做得完

### 主实验仍然保留 2×2

这个设计不用改，继续用：

* G1：Raw Review RAG + Base
* G2：Aspect-KB RAG + Base
* G3：Raw Review RAG + PEFT
* G4：Aspect-KB RAG + PEFT

这四组就足够回答两条研究问题。

### 辅助实验只留 1 个

我建议你只保留一个最值钱的消融：

**Fixed-Evidence Generation Test**
给 Base 和 PEFT **完全相同的候选和证据**，只比较：

* 槽位遵守
* 是否乱编
* 推荐解释组织
* 谨慎表达

这样你才能真正证明“PEFT 改善的是行为能力，而不是知识记忆”。

### 评测指标不要贪多

LangSmith 的 RAG 评测教程把 RAG 评估拆成 dataset、运行、然后按 answer relevance / answer accuracy / retrieval quality 来看；Ragas 目前稳定文档里提供了 Context Precision、Context Recall、Faithfulness、Response Relevancy 等一系列适合 RAG 的指标。另一方面，近期 CRS 评估工作指出，单靠命中率和规则指标很难真实反映用户体验，而像 ChatCRS 这类近期工作也强调外部知识和目标引导对推荐准确率与语言质量的重要性。([LangChain Docs][9])

所以你的指标体系压缩成下面这套最合适：

**RQ1（知识组织）**

* Recall@5 / nDCG@5
* Context Precision
* Faithfulness
* Citation Precision
* Citation Coverage

**RQ2（PEFT 行为）**

* Slot F1
* Numeric Constraint Accuracy
* Clarification Need Accuracy
* Clarification Target Accuracy
* Constraint Satisfaction Rate
* Unsupported Claim Rate

**人工主观评测**

* Trustworthiness
* Explanation Sufficiency
* Naturalness

其中人工评测只做 **20–30 个 case 的成对比较**，两位同学打分就够，你负责仲裁冲突样本。

## 二、数据量怎么取最合适

这些字段在 business schema 中是通用 business 数据的一部分，但与你当前论文主线关系很弱。([Datafiniti Docs][4])

## 五、具体清洗步骤：这部分是最关键的

### 第一步：空值与基本合法性过滤

建议直接删除：

* `name` 为空
* `city` 为空
* `reviews.text` 为空
* `reviews.rating` 为空
* `reviews.rating` 不在 1–5

因为 notebook 示例中最常用的也是酒店名、位置和评论评分文本几类字段。([Kaggle][3])

### 第二步：去重

你至少要做两层去重：

1. **酒店去重**
   同名酒店可能因地址写法差异重复出现，必须基于
   `name + city + province + postalCode (+ address)`
   进行规范化合并。

2. **评论去重**
   同酒店内如果 `title + text` 完全相同，删重。
   若 `text` 相同但时间不同，可保留一条并标记重复来源。

### 第三步：评论正文去噪

这个数据集里最需要处理的噪声，不是 HTML，而是**评论正文中混入的酒店管理者回复**。公开样例里就能看到，用户评论后面直接拼接了类似 “Dear ... I would like to thank you ... General Manager ...” 的回复内容；如果你不删掉，后面的方面抽取、情感判断和证据召回都会被污染。([GitHub][5])

建议规则化去除：

* `Dear ...`
* `General Manager`
* `With Hospitality`
* `Thank you for choosing ...`
* `look forward to welcoming you back`

同时去掉：

* `... More`
* URL
* 重复空白
* 乱码标点
* 全大写营销语

### 第四步：语言与长度过滤

因为这批数据主语料是英文，建议统一保留英文评论；若存在极少量非英文评论，直接删掉，避免对方面标签体系造成噪声。样例可见评论主体为英文文本。([GitHub][5])

建议阈值：

* 少于 **15 个英文词** 或 **50 个字符**：删除
* 超过 **2,000 字符**：截断或标为长文本
* 标题为空可以保留，只要正文足够长

### 第五步：时间标准化

保留 `reviews.date` 作为评论时间，转成标准日期，并额外衍生：

* `review_year`
* `review_month`
* `recency_bucket`

不要把 `dateSeen` 当评论发布时间。官方 schema 已明确区分这两个字段。([Datafiniti Docs][4])

### 第六步：评分标准化

统一把 `reviews.rating` 转成 1–5 整数或浮点，不要混用字符串。
后续你可以派生两个版本：

* **原始五级评分**
* **三分类情感近似标签**

  * 1–2：负向
  * 3：中性
  * 4–5：正向

这个三分类标签可以作为弱监督起点，但论文里要说明：它只是**评论整体情绪近似**，不等于句子级方面情感真值。

### 第七步：英文分句

建议按英文句号、问号、感叹号加规则分句；把长评论切成句子级证据单元。
后续所有检索、方面分类、证据引用，尽量都在**句子级**完成，而不是整条评论级。

## 七、关于“预算字段”和“入住时间字段”，你要主动降级

这是这套公开数据最容易踩坑的地方。

* **入住日期**：不作为推荐排序核心变量，只作为对话槽位展示或直接移除

这会让你的系统定义更干净。

## 八、因为语料是英文，你的检索模型也要跟着改

对应地，你的数据规划里可以直接增加两列：

* `query_lang`
* `doc_lang`

并在实验中统一设定：

* 语料库语言：英文
* 用户交互语言：中文或英文二选一
* 若用中文交互，则检索使用多语言 embedding

## 三、城市到底怎么选

### 先说原则

**不要先主观指定城市，再去看数据够不够。**
正确顺序是：**先统计，再选城市。**

因为这套数据是美国酒店评论数据，Datafiniti 自己的 business data 教程就是按“收集美国酒店、按州分析酒店”来讲的；Kaggle 数据页也强调它适合做 state-by-state 分析。也就是说，它天然更适合你做“美国城市酒店评论知识库”的实验，而不是直接默认某几个城市一定最优。 ([Datafiniti Docs][3])

### 我建议的城市选择标准

你先对 Jun19 全量数据跑一个 city profile，统计每个城市：

* 酒店数 `n_hotels`
* 评论数 `n_reviews`
* 每家酒店平均评论数 `avg_reviews_per_hotel`
* 评论时间跨度 `date_span_days`
* 平均评分 `avg_rating`
* 评分标准差 `rating_std`
* 平均评论长度 `avg_review_len`
* 最大酒店评论占比 `max_hotel_review_share`

然后按下面的规则筛城市：

**入围阈值**

* `n_hotels >= 8`
* `n_reviews >= 150`
* `avg_reviews_per_hotel >= 10`
* `date_span_days >= 180`
* `max_hotel_review_share <= 0.30`

这五条的意义分别是：

* 有足够多酒店，推荐才像推荐系统
* 有足够多评论，RAG 才有证据可召回
* 单酒店评论不能过多，否则城市数据会被一两家酒店主导
* 有时间跨度，后面才能做“近期评论优先”

### 推荐的城市组织方式

#### 方案 A：最稳的一月版

**1 个主城市 + 1 个小型外部验证城市**

* 主城市：用于建知识库、跑 2×2 主实验
* 外部验证城市：只做少量跨城市泛化测试

这个方案最适合你当前时间。

#### 方案 B：更像论文的版本

**2 个主城市**

* 一个“商务/会展/城市旅游混合型”
* 一个“休闲/度假型”

这样用户需求会更丰富，但工作量会明显增加。

## 七、酒店评论筛选策略

这部分是你后面写代码最关键的地方。我建议你直接按下面的层级实现。

### 第一层：硬过滤

评论必须满足：

* `reviews.text` 非空
* `reviews.rating` 非空
* `reviews.date` 非空
* 评分可转数值，且落在合理区间
* 文本长度不少于 **50 个字符** 或 **8 个英文 token**

### 第二层：去重

删除：

* 同一酒店下 `title + text` 完全相同的评论
* 同一酒店下文本相似度极高的近重复评论
* 仅有极短句、无信息量的评论
  例如 “good”, “nice”, “ok stay”

### 第三层：噪声删除

公开样本里能直接看到一个很典型的问题：**用户评论正文后面混入了酒店管理者回复**，比如
“Dear ...”, “General Manager”, “With Hospitality ...”
这类内容如果不删，后面的方面抽取和证据召回会被严重污染。样本里还出现了 `... More` 这种截断噪声。你应在清洗阶段专门写规则去除这些片段。 ([GitHub][4])

### 第四层：句子切分

把每条评论切成英文句子，后续所有 ABSA、RAG、证据引用都尽量在**句子级**做，而不是整条评论级做。

### 第五层：保留评论元信息

对每条评论至少保留：

* `review_id`
* `hotel_id`
* `reviews.date`
* `reviews.rating`
* `reviews.title`
* `reviews.text_clean`
* `city`
* `name`

## 十、训练集 / 测试集怎么切

这一步很重要：
**按酒店切，不要按评论随机切。**

原因很简单。
如果同一家酒店的评论同时出现在训练和测试里，你的模型、排序器和召回器很容易“见过类似表达”，结果会虚高。

所以建议：

* 训练集：70%
* 验证集：15%
* 测试集：15%

切分单位是 `hotel_id`，不是 `review_id`。

如果你只做 RAG 和规则排序，也至少要保证：

* 评测 query 涉及的目标酒店
* 不和调参阶段高度重合
