## 第一章 绪论：研究定位与问题定义

### 本章导读
本章从研究对象、工作流定位与问题意识出发，说明本课题为何不是泛化的开放式智能体研究，而是一个**面向特定数据底座与可验证实验目标的有状态推荐工作流研究**。在此基础上，本章进一步给出研究问题、创新点、研究假设与边界条件，为后续工作流设计与实验设计建立统一口径。

### 研究背景与意义
酒店推荐场景具有明显的评论依赖特征，相较于仅依据星级、均分或价格等粗粒度结构化字段进行推荐，评论文本能够更充分地反映用户真实关心的方面性体验，例如位置交通是否便利、房间设施是否完善、服务是否稳定、夜间是否安静等。

然而，原始评论文本天然存在信息分散、表达冗余、主观性强和难以直接用于稳定排序的问题；若进一步将其直接交由通用大语言模型处理，又容易出现事实引用不稳定、检索依据不透明和工作流边界不清晰等问题。

在此背景下，本课题的研究意义主要体现在两个方面。其一，从方法上看，课题尝试将酒店评论从非结构化文本转化为方面化、可检索、可聚合、可验证的知识底座，从而为可解释推荐提供更稳定的证据基础。

其二，从工作流上看，课题并不追求一个无限扩展的开放式智能体，而是研究在资源受限、时间受限且答辩导向明确的本科毕业设计场景中，如何构建一条知识外置、行为内化、证据可回查的有状态推荐工作流，这种定位既回应了推荐工作流与检索增强生成的结合需求，也使实验设计和论文论证更具可控性。

### 研究目标
围绕上述背景与问题意识，本文将研究目标概括为以下三个层次：

1. 构建一个以酒店评论为核心事实来源的可复用知识底座，包括结构化数据库、句子级证据索引、方面标签体系与酒店方面画像。
2. 设计并实现一个以偏好解析、候选排序、证据检索和推荐生成为核心环节的有状态会话式推荐工作流，使推荐结果能够落到真实评论证据之上。
3. 通过受控实验区分“知识组织方式”与“模型微调方式”的作用，形成能够直接支撑毕业论文方法章节与实验章节的研究结论。

### 1.1 课题一句话定义

本课题旨在构建一个**基于酒店评论知识库、方面感知检索与参数高效微调语言模型的有状态会话式酒店推荐工作流，并通过受控实验回答知识组织方式与微调方式对推荐可信度和会话行为质量的影响**。

### 1.2 工作流本质定位

从工程实现上看，本课题的工作流本质并不是一个开放式通用智能体，而是一个**以固定阶段流转为主、以结构化知识库为事实来源、以有限状态会话为交互外观**的推荐原型。因此，在实现框架上**更适合采用基于图式状态编排的workflow方案，而不是以工具自主调用为核心的开放式 agent loop**。

其基本运行链路可概括为：

```text
用户需求输入
  -> 偏好解析与槽位归一化
  -> 必要时触发澄清
  -> 候选酒店过滤与排序
  -> 句子级证据召回与重排
  -> 证据约束下的推荐生成
  -> 接收反馈并更新偏好
```

这种定位有三层意义：

1. 它保证工作流的每一步都能与数据库字段、向量检索结果和实验日志一一对应。
2. 它将“知识外置、行为内化”的设计原则贯彻到底，避免把酒店事实错误地塞进模型参数。
3. 它使项目更符合本科毕业设计的现实约束，即在有限时间与算力下优先实现**可验证、可复现、可答辩**的原型，而不是追求看似更完整但更不可控的产品形态。

### 1.3 两个核心研究问题

| 编号 | 研究问题 | 本质关注点 | 对应工作流部分 |
| --- | --- | --- | --- |
| RQ1 | 方面知识库画像引导的细粒度句子证据召回，是否优于未经方面引导的原始句子直接召回 | 知识组织方式如何影响检索质量、证据可验证性和推荐解释可信度 | 候选排序 + 证据检索 + 推荐生成 |
| RQ2 | 对基座模型进行偏好解析与澄清行为定向微调后，是否能显著提升槽位填充、澄清触发和约束遵守表现 | 微调是否主要提升行为层能力，而非事实记忆层能力 | 偏好解析 + 对话状态 + 推荐生成 |

### 1.4 课题的两个核心创新点

#### 创新点一：方面知识组织优于原始评论直接检索

本课题不只是将评论做成普通向量库，而是在句子级评论基础上进一步构建了方面标签、方面情感和酒店方面画像。

这样做的创新点不在于“多建了一张表”，而在于把评论从无结构文本转化为可统计、可排序、可按方面检索的知识单元，使推荐工作流能围绕用户真正关心的方面组织证据。

#### 创新点二：PEFT 的作用聚焦于会话行为而非事实知识

本课题明确区分了知识层和行为层：酒店事实通过知识库和检索模块提供，微调模型主要负责偏好解析、澄清交互、约束遵守和证据式表达。

这样做的意义在于让 RQ2 的实验结果更容易解释，也更符合小模型参数高效微调在资源受限场景下的真实优势。

### 研究假设
为了使研究问题能够在后续实验中被更清晰地检验，本文在正式实施前提出如下研究假设：

1. **H1**：在相同基础模型和相同评测集条件下，基于方面画像与方面感知过滤的检索组织方式**将优于**未经方面引导的原始句子直接检索方式，具体表现为更高的方面命中率、更强的证据可验证性与更稳定的推荐解释质量。

2. **H2**：在知识事实仍由外部知识库提供的前提下，PEFT 的主要收益将体现于偏好结构化解析、澄清问题触发、约束边界遵守与反馈更新等**会话行为层能力**，而非酒店事实层知识的“记忆”。

3. **H3**：当“方面知识组织”与“会话行为微调”同时存在时，工作流将在整体推荐链路上取得最优的综合表现，即 Aspect-KB Guided RAG 与 PEFT 的组合将构成本课题的完整工作流形态。

上述假设并不预设实验结论必须显著成立，而是为后续实验结果的解释提供清晰框架。即便部分假设仅得到有限支持，仍可据此分析问题究竟源于知识组织方式、训练样本构造还是当前数据边界本身。

### 2.5 当前向量检索底座

当前 ChromaDB 集合名固定为 `hotel_evidence`，持久化目录为 `DatafinitiHotelReviews/data/chroma_db`，向量编码模型为 `BAAI/bge-small-en-v1.5`。这意味着后续在线检索模块的语义召回源已经明确，不需要重新建立第二套并行索引。

同时还需要注意两个实现细节：

1. `evidence_index` 与 ChromaDB 的记录是一一对应的，因此最终引用必须落到真实 `sentence_id`。
2. 虽然离线方面标签允许一句多方面，但进入 Chroma 主元数据时，每个句子会被选取一个主标签用于过滤。这说明后续的方面感知检索应被准确描述为**基于主标签的 aspect-aware retrieval**，而不是完美多标签检索。

#### 2.5.1 中文交互与英文检索的桥接策略
鉴于当前评论主体为英文语料且向量编码模型采用英文嵌入模型，后续工作流不宜将中文原始输入直接作为最终检索查询写入向量检索层。

为降低跨语言语义偏移，本课题建议**采用“中文交互—结构化归一—英文检索表达”的桥接策略**：

首先由语言模型对用户中文输入完成偏好解析与槽位归一；其次根据 city、focus_aspects、avoid 与 hotel_category 等可执行字段生成一条简洁、可控的英文检索摘要；最后以该英文摘要结合 city、hotel_id、aspect 等元数据过滤条件完成句子级召回。

这一策略的意义在于用户交互层可以保持中文表达自然性，检索层则继续利用英文评论语料与英文嵌入模型的一致性优势。对于多轮场景，工作流应优先使用当前有效 WorkflowState 中的结构化偏好生成新的英文检索摘要，而不是直接拼接全部历史中文对话，从而减少检索噪声并提升跨轮一致性。

在论文表述上，该过程应被准确描述为“跨语言交互外观下的单语知识检索”，而不是泛化意义上的跨语言向量检索。这样既能够忠实反映当前技术路线，也有利于说明工作流边界与方法优势。

| 用途 | 建议选型 | 说明 |

### 2.6.4 模型职责边界与知识使用原则
后续生成与微调阶段**必须坚持“知识外置、行为内化”的职责分离原则**。

具体而言，酒店事实、评论证据、方面画像与排序信号均应由 PostgreSQL 与 ChromaDB 提供并接受回查，模型不承担酒店知识存储职责。

该原则**强调事实来源必须位于可查询、可校验、可追溯的外部知识库中，而不依赖模型参数隐式记忆**。

模型的主要作用应限定为四类：
- 其一，将用户自然语言偏好解析为受约束的结构化表示；
- 其二，在槽位不足、约束冲突或条件含混时触发必要澄清；
- 其三，在候选与证据已给定的前提下组织推荐说明；
- 其四，基于用户反馈执行偏好更新与响应调整。

该原则**强调模型微调的主要目标是提升会话行为质量，而非吸收酒店事实本身。**

这一职责边界对于本课题具有方法学意义。若将酒店事实直接寄希望于模型参数记忆，则工作流难以保证引用真实、解释可验证与错误可定位；反之，若坚持事实外置、模型只做行为决策，则推荐输出能够稳定回溯到真实 sentence_id 与结构化字段，RQ2 所观察到的提升也更容易被解释为偏好理解、澄清决策和约束遵守能力的提升，而非事实记忆偶然增强。

因此，后续 PEFT 的训练目标不应被表述为“让模型学会酒店知识”，而应被表述为“让模型更稳定地完成偏好解析、澄清触发、证据式表达与反馈更新等行为任务”。

### 2.7 酒店方面画像的唯一解释口径

当前 `final_aspect_score` 的唯一实现口径为：

```text
final_aspect_score = recency_weighted_pos - recency_weighted_neg - 0.3 × controversy_score
```

这一公式在后续项目中的地位非常关键：

1. 它是候选排序模块使用方面分数的唯一合法来源。
2. 它保证推荐工作流不仅能看正向证据，也能惩罚负向证据和高争议酒店。
3. 它为论文方法章节提供了明确、可解释的公式化规则，避免“排序由模型自由决定”的不透明叙述。

## 第三章 工作流总体方案设计

### 本章导读
在第二章明确数据基础与现实约束之后，本章进一步给出完整工作流方案。整体设计遵循“知识外置、行为内化、证据优先、链路可复现”的原则，将工作流拆解为数据层、偏好解析、候选排序、证据检索、对话状态管理、推荐生成、训练与评测八个模块，并说明它们在工程实现与论文写作中的双重作用。

### 设计原则
为保证工作流设计既符合工程可实现性，也符合学术论证的严谨性，后续模块设计统一遵循以下原则：

1. **知识外置原则**：酒店事实、评论证据与方面画像均以 PostgreSQL 和 ChromaDB 为唯一事实来源，不将其混入模型参数中。
2. **行为内化原则**：基础模型或 PEFT 模型主要承担偏好解析、澄清决策、证据组织与输出表达等行为性任务。
3. **证据优先原则**：任何推荐结论都必须能够回溯到真实 `sentence_id`，并接受引用有效性校验。
4. **边界诚实原则**：工作流必须显式记录并说明预算、精确距离、入住日期等未被当前数据底座正式支持的约束，避免出现能力幻觉。
5. **实验可复现原则**：关键配置、检索参数、数据切分与评测输入在实验阶段应保持冻结，以便研究结论具备可复核性。

### 3.0 技术框架选型与编排策略

尽管本课题**在工具层可采用 LangChain 生态**完成模型调用、提示模板管理、结构化输出解析、向量检索器封装与重排器接入，但**在工作流编排层**不宜将其直接实现为自由游走式开放智能体，而**应采用“LangGraph 负责状态编排，LangChain 负责组件封装”的双层实现策略**。

原因在于本课题的工作流**本质是一个具有固定阶段流转、有限状态更新和可回放日志的推荐工作流，其核心需求并非开放式工具自治，而是围绕偏好解析、澄清决策、候选排序、证据检索、推荐生成和反馈更新等节点的稳定流转**。

因此，工程实现上建议以图式状态机承载 WorkflowState，将各业务模块实现为节点函数，将澄清分支、推荐分支与反馈更新分支实现为条件边，从而保证状态更新路径、失败回退路径与实验日志路径均可被显式记录和复现。

在该实现口径下，**LangChain 的职责主要限于通用能力封装**，包括聊天模型客户端、结构化输出解析器、向量库检索器、reranker 接口与 prompt 模板管理等；**LangGraph 则承担全局流程编排、状态传递、条件路由和多轮对话控制职责**。

这样既能够复用 LangChain 的工程生态，又能够避免把本课题误实现为流程不可控的开放式智能体工作流。

### 3.1 工作流设计总图

项目整体可分为三层：

```text
事实层（Knowledge Base）
  PostgreSQL + ChromaDB + 中间产物

决策层（Ranking / Retrieval）
  偏好解析 -> 候选排序 -> 证据检索

交互层（Dialogue / Generation / PEFT）
  澄清 -> 推荐生成 -> 反馈更新 -> 行为对照实验
```

其中，事实层已经完成；决策层和交互层是后续实施主线；训练层与评测层则服务于研究问题验证和论文产出。

### 3.4 偏好解析模块设计

#### 3.4.1 模块目标

将用户自然语言表达压缩为当前知识库真正支持的结构化偏好对象，并识别但不承诺不支持的约束。

#### 3.4.2 输入

- 用户当前轮输入文本
- 现有会话状态中的历史偏好

若用户以中文输入，偏好解析模块需同时输出中文语义对应的结构化槽位与供检索模块使用的英文检索摘要。

### 3.5 候选排序模块设计

#### 3.5.1 模块目标

根据结构化偏好和酒店方面画像，从城市内酒店集合中筛选出一组可解释、可复现的候选列表，避免 LLM 在全库范围内直接自由发挥。

#### 3.5.2 输入

- 已确认的 `city`
- `focus_aspects`
- `avoid`
- 可选的 `hotel_category`

#### 3.5.3 输出

候选酒店列表 `HotelCandidate`，每个候选应包含：

| 字段 | 作用 |
| --- | --- |
| `hotel_id` / `hotel_name` | 酒店身份标识 |
| `score_total` | 最终排序分 |
| `score_breakdown` | 方面分、均分项、弱时间信号等分解 |
| `matched_aspects` | 命中的核心方面 |

#### 3.5.4 依赖底座

- `kb.v_hotel_overview`
- `kb.hotel_aspect_profile.final_aspect_score`
- `kb.hotel.avg_rating`

#### 3.5.5 在论文中的作用

它是 RQ1 中知识组织方式差异的上游实现基础，因为 Aspect-KB 路线将明显依赖方面画像，而朴素基线可减少或移除这一步。

#### 3.5.6 与其他模块的连接关系

排序结果进入证据检索模块，决定在哪些酒店上展开句子级证据召回。

### 3.6 证据检索模块设计

#### 3.6.1 模块目标

围绕候选酒店和用户关注方面，从 `hotel_evidence` 中召回真实句子证据，并通过 reranker 形成更高质量的证据包。

#### 3.6.2 输入

- 用户原始输入经偏好解析与英文检索摘要生成后的检索表达
- 候选酒店列表
- 目标方面
- 城市约束

#### 3.6.3 输出

`EvidencePack`，应至少包含：

| 字段 | 作用 |
| --- | --- |
| `hotel_id` / `hotel_name` | 所属酒店 |
| `query_text` | 当前查询内容 |
| `evidence_by_aspect` | 按方面分组的证据句 |
| `all_sentence_ids` | 平铺后的可引用句子 ID 列表 |

每条句子证据 `SentenceCandidate` 至少应包含 `sentence_id`、`sentence_text`、`aspect`、`sentiment`、`review_date`、`score_dense`、`score_rerank`。

#### 3.6.4 依赖底座

- `hotel_evidence` Chroma 集合
- `BAAI/bge-small-en-v1.5`
- `BAAI/bge-reranker-base`
- `kb.evidence_index` / `kb.v_evidence_full`

#### 3.6.5 在论文中的作用

该模块是 RQ1 实验的核心，因为证据质量、方面命中率和引用可验证性都直接建立在它的输出之上。

#### 3.6.6 与其他模块的连接关系

证据检索输出进入推荐生成模块，同时也会为日志、评测和案例分析提供直接素材。

#### 3.6.7 主标签检索局限与补充召回策略
需要指出的是，当前向量库主元数据为每个句子仅保留一个主方面标签，这虽然有利于实现高效的方面过滤，但也会引入多方面句被单标签化的问题。例如，同一句评论可能同时表达“位置便利”与“夜间噪声较大”，若仅按主标签硬过滤，部分本应进入证据集的句子可能被遗漏。

因此，后续检索模块建议采用“主通道 + 兜底通道”的两阶段证据召回策略。

主通道首先在候选酒店范围内执行基于主标签的 aspect-aware retrieval，以保持 RQ1 所需的知识组织主线；若某一目标方面在主通道中证据数不足、句子得分过低或证据极度同质化，则触发兜底通道，在相同候选酒店集合内部进行一次不加硬标签过滤的补充检索，并由 reranker 重新排序。

最终输出中应显式记录每条证据来自主通道还是兜底通道，以便后续分析主标签机制的收益与局限。如此既能保留方面知识组织的实验主线，也能避免系统因主标签单一化而在真实检索场景下表现过脆。

### 3.7 对话状态管理模块设计

#### 3.7.1 模块目标

把偏好解析、澄清、候选排序、证据检索和推荐生成组织成有限状态、多轮可回放的推荐工作流。

#### 3.7.2 输入

- 当前轮用户输入
- 既有 `WorkflowState`

#### 3.7.3 输出

更新后的 `WorkflowState`，建议包含：

| 字段 | 作用 |
| --- | --- |
| `turn` | 记录轮次 |
| `preference` | 当前有效偏好 |
| `pending_clarification` | 待澄清问题或缺失槽位 |
| `recommendations` | 当前轮推荐结果 |
| `feedback_history` | 用户反馈记录 |
| `conversation_history` | 原始对话轨迹 |

#### 3.7.4 依赖底座

它不直接定义底座事实，但必须遵守“可执行字段与不支持字段”的边界。

#### 3.7.5 在论文中的作用

该模块是会话式推荐的关键特征，也是 RQ2 中澄清触发准确率和反馈更新能力的主要载体。

#### 3.7.6 与其他模块的连接关系

它是唯一允许调用多个业务模块的编排层，负责把离散能力串成完整工作流。

### 3.8 推荐生成模块设计

#### 3.8.1 模块目标

在固定候选酒店和固定证据包条件下，生成可读、可验证、可记录的推荐解释，并对未支持约束做透明说明。

#### 3.8.2 输入

- `UserPreference`
- `HotelCandidate`
- `EvidencePack`

#### 3.8.3 输出

建议输出统一的 `RecommendationResponse`：

| 字段 | 作用 |
| --- | --- |
| `hotel_name` | 推荐对象 |
| `recommendation_text` | 面向用户的推荐说明 |
| `cited_sentence_ids` | 引用的真实句子 ID |
| `unsupported_note` | 对预算、距离、入住日期等未支持约束的说明 |

#### 3.8.4 依赖底座

- 证据句必须能回查到 `kb.evidence_index`
- 引用必须受当前 `EvidencePack` 约束

#### 3.8.5 在论文中的作用

推荐生成模块连接了工作流内部证据与用户可见输出，是解释可信度和证据可验证性的最终呈现层。

#### 3.8.6 与其他模块的连接关系

它消费排序和检索结果，并向评测模块和答辩演示提供最终可见产物。

### 3.9 训练模块设计

#### 3.9.1 模块目标

围绕偏好解析、澄清触发、证据式推荐和反馈更新构造监督数据，训练 PEFT adapter，并与基座模型开展行为对照。

#### 3.9.2 输入

- 冻结后的训练酒店集合
- 数据底座中的真实证据和结构化字段
- 人工构造或半自动构造的指令样本

#### 3.9.3 输出

- `preference_parse` 样本
- `clarification` 样本
- `grounded_recommendation` 样本
- `feedback_update` 样本
- adapter 权重、配置快照、训练日志

#### 3.9.4 依赖底座

- 训练样本必须只使用训练集合中的酒店与句子
- 不得将测试酒店证据或人工答案泄漏到训练集

#### 3.9.5 在论文中的作用

该模块是 RQ2 的实验基础，使“基座模型 vs PEFT 模型”的行为差异具有可追溯的来源。

#### 3.9.6 与其他模块的连接关系

训练模块的输出通过统一 LLM 客户端接入推理和实验流程，但逻辑上必须与在线工作流解耦。

#### 3.9.7 训练目标边界与防泄漏原则
为避免 PEFT 训练目标从“行为学习”漂移为“事实记忆”，训练数据构造必须遵守严格的职责边界。

原则上，preference_parse、clarification、constraint_compliance 与 feedback_update 应作为核心训练样本类型，其目标是提升结构化解析、澄清触发、边界表达与状态更新能力。grounded_recommendation 样本可以保留，但其职责应限定为学习“如何根据已给证据组织表达”，而不是学习具体酒店事实本身。

基于上述原则，训练集构造需额外遵守三条约束：
- 第一，训练样本中不得混入测试酒店的评论句、方面画像或人工编写的推荐答案；
- 第二，如使用酒店名称和评论证据构造推荐样本，其引用证据必须严格来自训练酒店集合；
- 第三，模型训练输入不应包含超出在线工作流可见范围的隐藏知识字段，以免形成离线训练时知道、在线推理时拿不到的能力落差。只有在此约束下，RQ2 的实验结果才能被合理解释为“行为层能力提升”，而非由知识泄漏带来的伪提升。

因此，本课题在训练阶段应优先扩大行为样本覆盖度，而不是盲目扩大酒店事实样本规模；若二者发生冲突，应优先保证行为样本的清洁性、边界性与可解释性。

### 3.10 评测模块设计

#### 3.10.1 模块目标

在冻结数据、冻结查询集和冻结 prompt 的前提下运行实验矩阵，计算 RQ1/RQ2 指标并输出论文材料。

#### 3.10.2 输入

- 冻结 split
- 冻结评测 query 集
- 槽位金标
- 四组实验输出

#### 3.10.3 输出

- RQ1 指标汇总
- RQ2 指标汇总
- 错误案例文件
- 实验报告与表格

#### 3.10.4 依赖底座

- `frozen_split_manifest.json`
- `judged_queries.jsonl`
- `slot_gold.jsonl`

#### 3.10.5 在论文中的作用

它直接对应实验章节，是项目结果转化为论文章节和答辩表格的关键桥梁。

#### 3.10.6 与其他模块的连接关系

评测模块只读取冻结资产与实验输出，不反向修改在线工作流逻辑。

## 第四章 项目实施路线与阶段任务

### 本章导读
本章采用更符合论文与项目协同推进需求的阶段式实施框架，每个阶段都围绕“目标、任务、交付物、验收标准、风险控制”展开，从而使后续研发活动与论文材料积累能够同步进行。

### 阶段推进原则
阶段推进过程中，应始终坚持以下控制原则：

1. 先冻结底座与边界，再扩展在线模块，避免在工作流实现中反复改动数据口径。
2. 先完成单轮可验证闭环，再增加澄清与反馈等多轮能力，避免对话状态过早膨胀。
3. 先完成可复现的最小实验矩阵，再考虑额外消融与展示包装，避免“功能看起来更多，但研究问题无法回答”。
4. 论文材料随阶段同步沉淀，而不是等待所有功能完成后再集中补写。

本项目后续实施不再按日或按周细拆，而采用**四阶段推进路线**。每个阶段都应具备明确的目标、任务、交付物、验收标准与风险提醒。

### 4.1 阶段一：数据底座总结与工程接口冻结

#### 阶段目标

将已完成的数据处理成果正式固化为后续工作流实现的唯一事实底座，并完成在线工作流所需接口和对象设计的冻结。

#### 核心任务

1. 整理数据底座统计结果、字段边界和知识库结构。
2. 明确在线工作流将消费的核心对象，如 `UserPreference`、`HotelCandidate`、`EvidencePack`、`RecommendationResponse`、`WorkflowState`。
3. 明确共享配置、数据库访问、Chroma 访问与 LLM 客户端的统一入口。
4. 冻结实验城市、方面体系、画像公式、`hotel_evidence` 集合等不可变约束。

#### 阶段交付物

- 本文档
- 统一的工作流设计草图
- 可供编码的核心对象和接口清单

#### 验收标准

- 读者能够清楚区分已完成底座与待实现模块。

### 4.2 阶段二：在线推荐闭环实现

#### 阶段目标

实现从用户输入到单轮可解释推荐输出的最小闭环，使工作流具备“能解析、能排序、能检索、能生成”的基础能力。

## 第五章 实验设计与评测方案

### 本章导读
本章以研究问题为牵引，对实验矩阵、评测集构造、指标体系与防污染原则进行统一规划。其核心目标不是单纯追求指标数量，而是保证实验设计能够直接回答 RQ1 与 RQ2，并支撑论文中“方法有效性”与“边界条件”两方面的论证。

### 实验设计目标
从论文方法论角度看，本课题实验设计需要同时满足三项要求：一是变量控制清晰，能够区分知识组织方式与微调策略各自的贡献；二是证据可回查，避免生成式结果无法落到真实评论依据；三是结论可解释，能够说明性能提升来自何处，性能不足又受到哪些现实因素限制。

### 5.1 主实验矩阵

本课题主实验采用 2×2 设计，行变量为知识组织方式，列变量为模型参数状态。

| 组别 | 知识组织方式 | 模型状态 | 实验目的 |
| --- | --- | --- | --- |
| G1 | Plain Sentence RAG | Base model | 基础基线 |
| G2 | Aspect-KB Guided RAG | Base model | 检验知识组织方式主效应 |
| G3 | Plain Sentence RAG | PEFT model | 检验微调主效应 |
| G4 | Aspect-KB Guided RAG | PEFT model | 完整工作流 |

### 5.2 两条研究问题如何落到实验上

| 研究问题 | 主要对比 | 观察重点 |
| --- | --- | --- |
| RQ1 | G1 vs G2；G3 vs G4 | 证据检索质量、方面命中、证据可验证性、引用真实性 |
| RQ2 | G1 vs G3；G2 vs G4 | 槽位填充、澄清触发、约束遵守、反馈更新与生成行为 |

#### 5.2.1 自变量控制与伪对照防护
为保证 RQ1 对知识组织方式的比较具有解释力，G1 与 G2、G3 与 G4 之间除知识组织方式外，不应同时变动其他关键环节。

具体而言，同一对照组必须共享相同的基础生成模型、相同的 prompt 模板、相同的向量库、相同的 top-k 参数、相同的 reranker 与相同的评测 query 集。组间唯一允许变化的核心因素，应限定为是否使用 hotel_aspect_profile 参与候选缩圈以及证据检索阶段是否采用 aspect-aware filtering 或 aspect-aware ranking。

如果组间同时改变候选排序逻辑、过滤字段、生成策略或重排配置，则实验结论将从“知识组织是否有效”退化为“两套工作流孰优”的粗糙比较，难以支撑 H1 的学术解释。因此，本课题应在实验前形成一份显式的组间冻结清单，并在实验报告中逐项记录未变化条件，以防出现伪对照。

换言之，本课题的 2×2 矩阵不只是实验分组表，更是一份严格的变量控制协议；只有在组间差异被约束到最小程度时知识组织方式的主效应才具有可信的解释空间。

### 5.3 评测集设计原则

评测集不追求大而全，而追求**小规模、强约束、可复核**。后续构造时应遵循以下原则：

1. 以冻结的酒店级 split 为前提，避免同酒店跨 train/test 泄漏。
2. query 需覆盖单方面偏好、多方面组合偏好和带不支持约束的输入。
3. 槽位金标应显式区分“可执行约束”和“识别但不执行约束”。
4. 所有人工评判记录都应可追溯到 `query_id`、`hotel_id` 与 `sentence_id`。

### 5.4 RQ1 指标总览

| 指标 | 作用 |
| --- | --- |
| Citation Precision | 检查模型声明的引用是否真实存在 |
| Aspect Recall@5 | 检查检索结果是否覆盖用户关注方面 |
| Evidence Verifiability | 检查引用句是否真正支持推荐理由 |

这些指标共同回答的问题是：**工作流是否检索到了相关、真实且能支撑推荐的证据**。

### 5.5 RQ2 指标总览

| 指标 | 作用 |
| --- | --- |
| Slot Precision / Recall / F1 | 评估偏好解析能力 |
| Constraint Compliance | 评估是否诚实处理未支持约束 |
| Clarification Trigger Accuracy | 评估是否在需要时正确发起澄清 |

这些指标共同回答的问题是：**模型是否更会理解需求、澄清信息并遵守边界**。

#### 5.5.1 指标操作化与人工标注规则
为避免指标定义停留在概念层，本课题需在正式实验前完成指标操作化。

对于 Citation Precision，不应仅校验 cited_sentence_ids 是否存在于数据库中，还应校验该句是否真实支撑生成文本中的对应推荐理由；若句子存在但不能支撑理由，则应记为无效引用。对于 Evidence Verifiability，建议采用人工标注的离散评分规则，例如 0 表示完全不支持，1 表示部分支持或证据过于模糊，2 表示明确支持且与目标方面一致。

对于 Clarification Trigger Accuracy，需先定义“应澄清”的判定边界，例如缺失核心城市槽位、用户表达包含多个相互冲突的核心方面或用户输入以当前系统不支持约束为主且不足以形成可执行查询时，应视为需要澄清；反之，若现有槽位已足以形成候选过滤与证据检索，则不应额外发起澄清。对 Constraint Compliance，也应明确区分“识别到但诚实声明不支持”与“未识别或错误承诺支持”两类行为。

通过上述操作化，RQ1 与 RQ2 的评测结果才具有可复核性和可比较性；否则，即使得到数值差异，也难以说明差异究竟来自知识组织方式、模型行为提升还是来自人工判断口径不一致。

因此，本课题在实验正式运行前，应先冻结指标定义、人工标注说明书与案例判定样例，再进入批量实验阶段。

### 5.6 防污染原则

后续实验必须严格遵守以下原则：

1. 评测集冻结优先于模型调参。
2. 同一组实验只能改变一个自变量。
3. 所有训练样本只能来自训练酒店集合。
4. prompt、top-k、向量集合、reranker 和 query 集在组间必须保持一致。
5. 不支持约束不能被包装成已支持能力。

### 5.7 实验结果应沉淀的材料

实验阶段结束后，至少应形成以下成果：

| 材料 | 用途 |
| --- | --- |
| 四组实验输出日志 | 复现和排查 |
| RQ1 指标汇总表 | 论文实验章节主表 |
| RQ2 指标汇总表 | 论文实验章节主表 |
| 成功案例与失败案例 | 论文案例分析与答辩展示 |
| 配置快照与模型路径记录 | 保证实验可复算 |

## 结语

综上所述，本课题当前已经完成了最难以返工的数据底座部分，并具备了向在线推荐闭环、PEFT 行为微调、2×2 实验矩阵和论文写作推进的坚实基础。

因此，后续项目推进应始终围绕三条并行主线展开：

1. **设计主线**：构建偏好解析、候选排序、证据检索、对话状态与推荐生成的在线闭环；
2. **实施主线**：按阶段补齐在线原型、PEFT 微调和实验运行；
3. **论文主线**：把项目产物工作流化映射到方法章节、实验章节与结论章节。
